01What CBM OS is

CBM OS is Juli's AI-native operating system — an Obsidian vault at ~/CBM OS/ that is the single source of truth for every project, client, person, decision, and piece of reusable knowledge across both CBM creative/technical work and GAP consulting deliverables.

The guiding principle is one line:

Operating philosophy"Own your ideas. Rent the AI." Everything Juli thinks lives in plain Markdown she controls. The AI tool (Claude today; Codex, Gemini, or local tomorrow) is a renter that points at the vault — never the owner of it.

Why this matters for you as staff: when you work a project, the vault is authority, not the workspace folder. Code, scripts, and .env files live in workspaces under ~/CBM/ and ~/GAP/ — but context, status, and decisions live in the vault. If a workspace's notes disagree with the vault, the vault wins.

02The three-layer architecture

The whole system is deliberately split into three layers so it stays AI-tool-agnostic. The bottom layer is Juli's thinking, the middle layer translates it for any AI, and the top layer is whichever AI tool is plugged in this week.

The key design move: no AI-tool-specific logic ever goes in the bottom two layers. That's how the OS ports cleanly to a different AI later without rewriting the knowledge base.

04★ The Airtable PAT model read carefully

This is the part to internalize before you touch any client data. CBM runs many Airtable bases across many clients, and the security model is built on one non-negotiable idea:

The rule that governs everythingCredentials never cross projects. One project's Personal Access Token (PAT) is never another project's PAT. Treat every project as if it were a different company — because for credential purposes, it is.

Where credentials live (and don't)

There are two physically separate places, and keeping them separate is the whole game:

LocationHoldsSecrets?
The vault — CBM OS/Projects/<type>/<slug>/The pointer.md (context, deliverables, status) and a .env.example template only, with placeholders like pat_REPLACE_ME.NO — ever
The workspace — <workspace>/.claude/.envThe real Airtable base ID + PAT for that one project. Gitignored. Read only by that workspace's own scripts.YES — only here

The vault is readable by any AI that points at it, so no real secret is ever allowed in the vault. The real .env sits in the live workspace, is listed in .gitignore, and never gets committed or copied back into the vault.

How a project records its base (without the secret)

The authoritative index Projects/_projects.json stores each project's non-secret airtable_base_id (e.g. appXXXXXXXXXXXXXX) so routing and tooling know which base to hit. The PAT itself is never in JSON — only the base ID, which is not sensitive on its own. The PAT lives exclusively in the workspace .env.

What a PAT should actually be scoped to

When you create a PAT in Airtable, scope it to the minimum that the project needs:

  • Do grant access to only that client's base(s) — never "all current and future bases."
  • Do grant only the scopes the work needs (typically data.records:read, data.records:write, and schema.bases:read).
  • Don't reuse one PAT across two clients to "save time." That's the one thing this whole model exists to prevent.

05★ PAT setup — step by step

This is the exact path to wire a project's Airtable credentials. Do this once per project workspace.

Step 1 — Create the scoped PAT in Airtable

  1. Go to airtable.com/create/tokens → Create token.
  2. Name it for the project, e.g. cbm-<client>-readwrite.
  3. Add scopes: data.records:read, data.records:write, schema.bases:read.
  4. Under Access, add only that client's base(s).
  5. Copy the token (starts with pat...). You only see it once.

Step 2 — Find the base ID

It's already in Projects/_projects.json under the project's airtable_base_id. Or open the base in your browser — the URL contains app... (that's the base ID).

Step 3 — Create the workspace .env from the template

Every workspace ships a template. Copy it to the real (gitignored) location and fill it in:

# from inside the project workspace
cp .claude/.env.template .claude/.env

# then edit .claude/.env to contain the real values:
AIRTABLE_BASE_ID=appXXXXXXXXXXXXXX
AIRTABLE_PAT=pat_your_real_token_here

Step 4 — Confirm it's gitignored

Every workspace .gitignore already lists these. Confirm before your first commit:

.claude/.env
.claude/settings.local.json
.env
*.pat
Sanity checkRun git status after creating .env. If .claude/.env shows up as a tracked/untracked file Git wants to add — stop and fix the .gitignore before doing anything else.

Step 5 — Use the credentials from a script

Two supported ways to load the env. Python uses the shared load_env.py helper (in each workspace's 05 - Scripts/):

from load_env import load
load()
PAT = os.environ["AIRTABLE_PAT"]
BASE = os.environ["AIRTABLE_BASE_ID"]

Shell / curl sources the file directly:

set -a; source .claude/.env; set +a
curl "https://api.airtable.com/v0/$AIRTABLE_BASE_ID/Table" \
  -H "Authorization: Bearer $AIRTABLE_PAT"
Why the helper existsload_env.py reads ../.claude/.env relative to the script and uses setdefault — so a value already in the environment wins and nothing leaks into logs. Import it; don't paste keys into the script.

If a secret ever leaks

Incident response — do all three
  1. Remove it from wherever it landed immediately.
  2. Rotate the credential — regenerate the PAT in Airtable so the leaked one is dead.
  3. Log the incident in Atlas/Decisions/ so we have a record.

06★ Versioning & backup of the OS

The whole vault is a flat tree of plain-text Markdown — which is exactly what makes it cheap to version and bulletproof to recover. We run versioning at two complementary levels.

Level 1 — Live versioning via Obsidian Sync

Obsidian's core sync plugin is enabled on the vault. This gives continuous, real-time versioning: every save is propagated, and Sync keeps per-file version history you can roll back to from inside Obsidian. This is the "undo my last hour" layer.

Note for staffYou generally read the vault, not edit it directly — most writes happen through the AI following the vault-map.md rules. If you do edit in Obsidian, Sync history is your safety net.

Level 2 — Versioned off-site archive via backup-to-drive.sh

The deeper safety net is Scripts/backup-to-drive.sh. It mirrors the vault to Google Drive with rclone — and crucially, it's a versioned mirror, not a destructive one:

  • Current state syncs to gdrive:CBM-OS-Backup.
  • Anything deleted or overwritten is preserved under gdrive:CBM-OS-Archive/<timestamp>/ via rclone's --backup-dir.
  • An exclude list (Scripts/.backup-exclude.txt) keeps junk out; every run is logged to Scripts/.backup.log.
# mirror current vault, archive anything changed/removed
rclone sync "$VAULT" gdrive:CBM-OS-Backup \
  --exclude-from .backup-exclude.txt \
  --backup-dir gdrive:CBM-OS-Archive/$(date +%Y%m%d-%H%M%S) \
  --log-file .backup.log --log-level INFO
Why this design is "optimized"The two layers cover different failure modes with zero overlap. Sync = fast, per-file undo while you work. rclone archive = immutable, timestamped, off-site point-in-time recovery — so a bad bulk operation, a sync conflict, or a lost machine never loses history. Plain Markdown means every archived version is human-readable forever, with no proprietary lock-in. That's the "own your ideas" principle made operational.

What is deliberately NOT versioned

Not versioned / not backed upWhy
Workspace .claude/.env files (they live outside the vault)Secrets never enter the vault, so they never enter Sync or the Drive archive. By design.
Anything in .backup-exclude.txt (.DS_Store, caches, etc.)Noise that would bloat the archive.
CorollaryBecause secrets are never versioned, losing a workspace means re-creating its .env — the PAT is rotated/re-pasted, not restored from backup. That's the safe tradeoff: no secret is ever sitting in a backup waiting to leak.

07New-staff onboarding checklist

  1. Get vault access. Juli adds you to Obsidian Sync for the CBM OS vault (read-oriented).
  2. Read the three core files once yourself: AIOS/me.md, AIOS/vault-map.md, AIOS/skill-map.md. They are the contract.
  3. Learn the routing. Projects/_projects.json is the authoritative project index — path, slug, slash command, status, deadline, base ID for every workspace.
  4. Pick up your project workspace under ~/CBM/ or ~/GAP/.
  5. Wire its Airtable PAT following Section 05 — scoped token, real .env from template, confirm gitignored.
  6. Verify the backup path is healthy if you own infra: rclone listremotes shows gdrive:, and Scripts/.backup.log shows recent successful runs.
  7. Reference deliverables by number on GAP work (e.g. "<Project> #34"), always.

08Hard rules (never break)

Non-negotiables
  1. Never commit, paste, or surface a credential. All secrets stay in workspace .claude/.env files — never in the vault, JSON, settings, or scripts.
  2. Per-project credentials don't cross projects. One PAT, one client.
  3. The vault is the source of truth. pointer.md wins over any workspace CLAUDE.md on disagreement.
  4. Stay out of Raw/ — immutable source material. Cite, don't mutate.
  5. Don't auto-organize Inbox/ — that's Juli's via /process-inbox.
  6. Cite GAP deliverables by number. Non-negotiable for billable clarity.
